Authentication Explained:
How Login Systems Work

• Every app asks you to log in. But do you actually know what is happening behind the scenes? Let's break down Authentication from zero. 👇

Step 01 / Introduction

Understand the Core

Authentication is one of those concepts every developer uses...

But many don't truly understand until they build it themselves. The good news? The idea is actually simple.

Step 02 / Definition

🔐 The Core Question

Authentication answers one question:

"Are you really who you say you are?"

When you enter your email/password and click Login, the application is trying to verify your identity.

Step 03 / Distinction

Authentication != Authorization

A common beginner mistake is confusing these two concepts. They are NOT the same thing:

  • Authentication: Who are you?

    This gets you through the front door of the system.

  • Authorization: What can you do?

    Decides which rooms and endpoints you are allowed to enter.

Step 04 / Credentials

Sending Credentials

You enter your email and password on the page. The frontend sends those credentials to the server, and the server begins investigating.

Step 05 / Database Check

Searching the User

First, the server queries the database:

If No Email Exists ❌ Login Fails

Stop execution immediately and respond with 401 Unauthorized.

If Email Exists ✅ Continue

Proceed to verify the password credentials.

Step 06 / Passwords

Never Store Raw Passwords

Here is a rule every single developer must know:

Your password is NOT stored in the database. At least, it shouldn't be.

Step 07 / Security

🔒 Storing Passwords as Hashes

Instead of plaintext, the database stores a **cryptographic hash** — a scrambled, unreadable version of the password:

# Plaintext password123 # Encrypted Hash $2b$10$7Y2kPxM9q...

Hashing functions are strictly **one-way**. You cannot reverse a hash back into the original password.

Step 08 / Verification

Comparing Hashes

When you attempt to log in:

  • Hashing input

    Server takes the password you typed and passes it through the same hashing function.

  • Matching hashes

    Compares this new hash with the database hash. If they match: Login successful.

Step 09 / HTTP Protocols

The Stateless Problem

Now the server knows: *"Yes, this is Harry."*

But a new problem appears: How does it remember you on your next requests?

Step 10 / Statelessness

HTTP has No Memory

Every single request is treated like a brand-new conversation. The server forgets who you are immediately after sending a response.

⚠️ We need a way to remember user states.

Step 11 / Sessions

🍪 Cookie-Based Sessions

The traditional solution:

After login, the server creates a Session object and stores it in memory or a database. It then dispatches a unique **Session ID** back to the browser inside a Cookie.

Step 12 / Handshakes

Automatic Identity Matching

Now, every single outgoing browser request automatically attaches the cookie:

Cookie: session123

The server inspects the session ID, finds your logged-in profile details, and keeps you logged in.

Step 13 / Sessions Cost

The Downside of Sessions

The server must store and track every active session lookup.

As applications scale to millions of concurrent users, managing this database state becomes extremely memory-intensive.

Step 14 / Modern Tokens

📦 JSON Web Tokens (JWT)

Modern applications solve memory overhead using JWTs.

Instead of storing identity parameters on the server, user identity is stored directly inside the token itself on the client.

Step 15 / Token Generation

Token Signature

After login, the server creates a token, signs it using a cryptographic secret key, and sends it back to the client:

# JWT Structure eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
Step 16 / Token Handshakes

The Authorization Header

The client stores the token and sends it manually with every request:

Authorization: Bearer TOKEN

The server parses the token, verifies the cryptographic signature, and immediately trusts your identity without checking a session database.

Step 17 / Comparison

Stateful vs. Stateless

🍪 Sessions Stateful

Identity is stored on the Server. Client carries the ID.

📦 JWT Stateless

Identity is stored on the Client. Server verifies signatures.

Step 18 / Social Logins

🌐 OAuth Integration

OAuth is what powers social login buttons:

"Login with Google • Continue with Facebook • Sign in with GitHub"

Step 19 / Trust Delegation

Federated Trust

Instead of handling passwords yourself, you delegate validation to a trusted provider:

User logs into Google → Google confirms identity → Google sends token verification to your app → App logs you in.

Step 20 / Security Boundary

No Password Leakage

Your application never sees Google, Facebook, or GitHub passwords. The provider handles authentication, and your application simply receives confirmation.

SUMMARY

Authentication Summary

  • Authentication = Proving Who You Are 🔐
  • Authorization = Access permissions 🚪
  • Sessions = Server-side memory storage 🍪
  • JWT = Client-side stateless token 📦
  • OAuth = Social validation delegation 🌐
CONCLUSION

Ready to Build

"Once you understand Authentication, Sessions, JWTs, and OAuth... you are ready to build."

Every login system is built on these core pillars. Master authentication to make backend engineering start getting really interesting.
